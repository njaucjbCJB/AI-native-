# AI 企业运营框架控制台 PRD

## 1. 问题陈述

当一位 CIO 或技术负责人进入一家新公司时，通常会面对大量分散、重复、低效率的内部业务流程，例如采购审批、报销审批、请假审批、合同审批、招聘审批等。

这些流程往往存在以下问题：

- 表单和审批系统各自独立，难以复用。
- 业务数据散落在 Excel、OA、聊天工具、邮件和不同 SaaS 系统里。
- 审批规则写死在系统里，调整成本高。
- 数据归档不统一，后续审计、查询、分析困难。
- 报表依赖人工整理或固定 BI 模板，无法按管理者问题动态生成。
- AI 能力通常只是外挂聊天助手，没有真正进入表单、流程、数据和决策链路。
- 企业想引入 AI CEO、AI 管理助手或 Agent 系统时，缺少一套可落地、可演进的技术框架。

用户的目标不是先做一个单独的采购审批应用，而是先验证一套 **AI-native 企业业务系统框架**：

> 业务需求可以被 AI 转换成业务蓝图，业务蓝图驱动表单、审批流、数据沉淀、归档、报表和 Agent/Skills 调用。

### 1.1 核心目的：让 AI 和人共享业务上下文

这套系统不是在传统业务系统外增加一个聊天窗口，而是让 AI 和人共同成为业务流程的参与者。AI 是系统的一等用户，也是最重要的使用者之一。

传统系统通常把数据、规则和流程分散在不同页面、服务和接口中。人可以依靠组织经验理解这些边界，但 AI 往往需要调用大量碎片化接口，再重新拼接上下文，仍然难以判断数据含义、当前流程阶段、适用规则和历史原因。

本系统要把业务上下文组织成 AI 可以快速连接和理解的统一结构：

- 数据说明当前发生了什么。
- 规则说明为什么允许、拒绝或需要升级处理。
- `Workflow` 说明业务当前处于哪个阶段，以及下一步可以执行什么动作。
- AI 在权限范围内参与生成、分析、建议、审批辅助和复盘。
- 反馈记录业务结果和人工修正，用于后续优化规则与 AI 行为。
- 审计记录保存数据变化、流程动作、AI 建议和人工决定，使系统行为可以追溯和解释。

因此，系统的核心控制回路是：

```text
数据 + 规则 + AI + Workflow → 业务动作 → 人工/系统反馈 → 审计记录 → 新的业务上下文
```

第一版产品要做成一个面向 CIO/技术负责人的 demo，用最小开发量证明这套框架可以跑通，并为后续真实数据库、真实 AI、真实工作流引擎和企业系统集成留下清晰替换路径。

## 2. 解决方案

建设一个 **AI 企业运营框架控制台**。

第一版是一个可部署的静态 Web demo，使用 Vite、React、TypeScript 和 LocalStorage 实现，不依赖真实后端、真实数据库或真实 AI API。

产品采用“框架控制台 + 采购审批样例流程”的形态：

- 框架控制台用于展示底层能力：业务蓝图、数据模型、表单 Schema、审批流、Agent/Skills、运行实例、AI 报表、架构替换路线。
- 采购审批样例用于证明业务闭环：生成蓝图、部署蓝图、提交采购申请、AI 模拟风险分析、AI 模拟审批路由、人工模拟审批、数据归档、报表生成。

系统的核心不是固定页面，而是四个可替换抽象：

- `Blueprint`：业务蓝图，是表单、流程、规则、归档和报表的中心契约。
- `StorageAdapter`：数据存储适配器，第一版使用 LocalStorage，后续替换 PostgreSQL 或 Supabase。
- `AgentRuntime / SkillRegistry`：AI 与 Skills 执行层，第一版使用 MockAgentRuntime，后续替换真实大模型和工具调用。
- `WorkflowRuntime`：轻量流程运行时，第一版支持提交、审批、拒绝、条件路由、归档，后续可替换 Flowable、Camunda 或 Temporal。

这些抽象必须共同提供统一的业务上下文，而不是形成彼此隔离的信息边界。界面用户、Workflow 和 AI 应在权限控制下使用同一套业务对象、规则定义、流程状态和审计证据。

第一版成功后，CIO/技术团队应能清楚判断：

- 这套框架能否从业务需求生成可运行流程。
- 哪些模块适合继续自研。
- 哪些模块应该接入开源工具或商业系统。
- 后续如何从 demo 演进为真实企业级系统。

## 4. 实现决策

### 4.1 产品形态

- 第一版采用“框架控制台 + 采购审批样例流程”。
- 控制台展示框架能力，采购审批用于证明运行闭环。
- 目标用户是 CIO、技术负责人和企业架构负责人，不是普通业务用户。
- 产品重点是技术框架可行性，而不是生产级业务系统完整度。

### 4.2 技术栈

- 使用 Vite、React、TypeScript 构建静态前端 demo。
- 使用 LocalStorage 保存 demo 数据。
- 不接真实后端、真实数据库、真实 AI API、真实工作流引擎。
- demo 要能构建为静态网页，后续可部署到 Netlify、Vercel 或 GitHub Pages。

### 4.3 核心深模块

第一版需要优先构建以下深模块。深模块指封装大量功能但对外接口简单、稳定、可独立测试的模块。

- `Blueprint`：业务蓝图契约，承载数据模型、表单、流程、规则、归档、报表定义。
- `StorageAdapter`：统一持久化边界，第一版实现 LocalStorage 版本。
- `AgentRuntime`：统一 AI/Skill 调用边界，第一版实现 MockAgentRuntime。
- `SkillRegistry`：声明和管理可用 Skills。
- `WorkflowRuntime`：统一流程状态推进边界，第一版实现轻量流程运行时。
- `ReportGenerationSkill`：从运行数据生成指标和 AI CEO 摘要。

### 4.4 控制台模块

第一版控制台包含 8 个视图：

- Blueprints / 业务蓝图
- Data Model / 数据模型
- Form Schema / 表单定义
- Workflow / 审批流
- Agent & Skills / Agent 与技能注册表
- Runtime / 运行实例
- Reports / AI 报表
- Architecture / 架构与替换路线

### 4.5 Blueprint 决策

Blueprint 是系统中心契约，必须包含：

- 元信息
- 数据模型
- 表单字段
- 角色权限
- 审批流程
- 风险规则
- 归档策略
- 报表定义

第一版由 MockAgentRuntime 根据默认采购需求生成采购审批 Blueprint。

### 4.6 采购审批样例决策

第一版采购表单字段包括：

- 采购物品
- 部门
- 金额
- 供应商
- 期望到货日期
- 采购理由

默认部门包括：

- Engineering
- Sales
- Operations
- Finance

默认供应商白名单包括：

- Apple
- Dell
- Lenovo
- Amazon Business

### 4.7 风险规则决策

第一版风险来源包括：

- 金额大于等于 10000。
- 供应商不在白名单。
- 30 天内存在相似采购。
- 采购理由过短或信息不完整。

风险等级计算：

- 命中 0 条：低风险。
- 命中 1 条：中风险。
- 命中 2 条或以上：高风险。

### 4.8 审批流决策

第一版审批路径规则：

- 所有申请先进入部门经理审批。
- 金额大于等于 10000 或风险为中/高时，进入财务审批。
- 高风险申请进入 CEO 确认。
- 审批完成后自动归档。
- 任一审批节点拒绝后，申请进入拒绝状态并归档。

### 4.9 Agent/Skills 决策

第一版 Skills 包括：

- `RequirementParsingSkill`
- `BlueprintGenerationSkill`
- `RiskAnalysisSkill`
- `ApprovalRoutingSkill`
- `WorkflowExecutionSkill`
- `ArchiveSkill`
- `ReportGenerationSkill`

每次 Skill 调用都要生成 Agent Activity，包含：

- Skill 名称
- 输入摘要
- 输出摘要
- 执行状态
- 时间

### 4.10 数据持久化决策

- UI 组件不得直接读写 LocalStorage。
- 所有数据读写必须经过 StorageAdapter。
- LocalStorage key 需要命名空间，例如 `aiof.blueprints`、`aiof.requests`、`aiof.activities`。
- 刷新页面后，Blueprint、申请实例、审批历史、归档和 Agent Activity 必须保留。

### 4.11 UI 决策

- 使用左侧导航展示框架模块。
- 使用顶部角色切换模拟不同角色。
- 使用中间主工作区展示当前模块。
- 使用右侧 Agent Activity 面板展示 AI/Skill 活动流。
- 角色切换只用于 demo，不代表真实认证和权限系统。

### 4.12 架构替换路线决策

第一版必须展示 mock 到生产的替换路线：

- LocalStorageAdapter 替换为 PostgreSQL 或 Supabase。
- MockAgentRuntime 替换为真实 OpenAI、本地模型或多 Agent 运行时。
- Mock Skills 替换为真实工具调用 Skills。
- LightweightWorkflowRuntime 替换为 Flowable、Camunda 或 Temporal。
- Mock 通知替换为邮件、飞书、企业微信或 Slack。
- 静态报表替换为 SQL、BI 和 AI 报表生成。

## 5. 测试决策

### 5.1 测试原则

- 测试外部行为，不测试内部实现细节。
- 优先测试深模块，而不是测试 UI 的每一个展示细节。
- 测试应围绕业务闭环和模块契约设计。
- Mock 数据应固定，保证测试可重复。
- UI 测试只覆盖关键用户路径。

### 5.2 需要测试的模块

- Blueprint 生成：输入默认采购需求后，生成包含数据模型、表单、流程、风险、归档、报表的 Blueprint。
- StorageAdapter：保存和读取 Blueprint、申请实例、Agent Activity，刷新后数据不丢失。
- RiskAnalysisSkill：不同金额、供应商、重复采购、理由长度组合能得到正确风险等级。
- ApprovalRoutingSkill：低、中、高风险能生成正确审批路径。
- WorkflowRuntime：创建申请、审批、拒绝、归档状态推进正确。
- ReportGenerationSkill：基于申请数据生成正确指标和摘要。
- Agent Activity：关键 Skill 调用都能产生可追踪日志。

### 5.3 关键测试场景

1. 默认采购需求可以生成 Blueprint。
2. 部署 Blueprint 后，运行区可以渲染采购申请表。
3. 必填字段为空时不能提交。
4. 低风险申请只需要部门经理审批。
5. 金额大于等于 10000 的申请需要部门经理和财务审批。
6. 高风险申请需要部门经理、财务和 CEO 审批。
7. 审批通过后申请归档。
8. 任一节点拒绝后申请归档为拒绝状态。
9. 报表能展示采购总金额、状态分布、高风险数量。
10. Agent Activity 面板能显示 Blueprint 生成、风险分析、审批路由、归档和报表生成。
11. 刷新页面后，已部署 Blueprint 和运行数据仍然存在。
12. 重置 demo 数据后，LocalStorage 中相关数据被清空。

### 5.4 第一版测试优先级

最高优先级：

- Blueprint 生成
- RiskAnalysisSkill
- ApprovalRoutingSkill
- WorkflowRuntime
- StorageAdapter

中等优先级：

- ReportGenerationSkill
- Agent Activity
- 动态表单提交

低优先级：

- 纯视觉布局
- 图标展示
- 静态说明文案

## 6. 不在第一版范围内

第一版不做以下能力：

- 真实登录。
- 真实数据库。
- 真实 AI API。
- 真实文件上传。
- 真实工作流引擎。
- 真实邮件、飞书、企业微信或 Slack 通知。
- 多租户。
- 多公司隔离。
- BPMN 流程设计器。
- 复杂权限矩阵。
- 外部 ERP、CRM、OA 集成。
- 自然语言 SQL。
- 真实审计合规能力。
- 生产级安全策略。
- 多业务模板批量生成。
- 普通业务用户使用的完整低代码编辑器。

这些能力后续可以逐层加入，但第一版必须聚焦框架可行性。

## 7. 进一步说明

第一版不是为了证明“采购审批应用能不能做”，而是证明：

> CIO 可以用这套 AI-native 框架，把业务需求快速转成可运行的企业流程系统。表单、审批、数据、归档、报表、Agent/Skills 全部通过统一框架协作。

最重要的产品判断：

- 业务蓝图是中心。
- 数据沉淀是底座。
- Agent/Skills 是能力层。
- WorkflowRuntime 是流程骨架。
- 控制台是 CIO/技术团队理解和实施系统的入口。

第一版完成后，下一步不应立即扩展更多业务流程，而应该先评估：

- Blueprint 契约是否稳定。
- 深模块边界是否清晰。
- Mock 层替换为真实组件是否顺畅。
- 控制台是否能让技术负责人理解整套架构。
- 采购审批样例是否足以证明表单、流程、数据、归档和报表闭环。
