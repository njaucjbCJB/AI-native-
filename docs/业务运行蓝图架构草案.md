# 业务运行蓝图架构草案

本文档描述产品从“动态表单与审批流程”升级为“AI-native 企业业务运行框架”的目标架构。它是未开发蓝图，用于对齐业务架构、应用架构、数据架构和产品核心竞争力。

## 1. 业务架构

产品的业务起点不是表单，而是一个自然语言表达的业务目标。系统先判断目标应该如何被完成，再生成对应的业务运行蓝图。

```mermaid
flowchart TD
  A["业务目标 / 项目事务<br/>非结构化自然语言输入"] --> B["目标接收<br/>Intent Intake"]
  B --> C["任务分发技能<br/>Task Routing Skill"]

  C -->|可自动完成| D["自动执行路径<br/>Autonomous Execution"]
  C -->|需要多人配合| E["协同流程路径<br/>Collaborative Workflow"]
  C -->|AI 先处理，人再确认| F["混合路径<br/>Hybrid Execution"]

  D --> D1["调用 Skill"]
  D1 --> D2["生成交付物"]
  D2 --> G["AI 验收模块"]

  E --> E1["生成表单"]
  E --> E2["生成人工任务"]
  E --> E3["生成审批流"]
  E --> E4["生成权限和数据隔离规则"]
  E1 --> H["Workflow 执行"]
  E2 --> H
  E3 --> H
  E4 --> H
  H --> G

  F --> F1["AI 预处理 / 生成初稿"]
  F1 --> F2["人工确认 / 补充 / 审批"]
  F2 --> G

  G --> I["业务实例"]
  I --> J["数据库"]
  J --> K["报表 / 审计证据 / AI 可连接上下文"]
```

### 业务架构的关键判断

- 表单不是入口，业务目标才是入口。
- 审批流不是默认路径，只有需要多人协作时才生成。
- Skill 不是辅助功能，而是自动执行路径的主体。
- AI 验收是统一出口：无论自动执行还是人工协同，都需要形成验收结果。
- 最终沉淀的不是一张表单，而是一个可查询、可审计、可复盘的业务实例。

| 模块 | 职责 | 当前状态 |

| Task Routing Skill | 判断自动执行、协同流程或混合路径 | 需要开发 |

数据架构以业务实例为中心。Business Operating Blueprint 是规则和契约，Business Instance 是运行后的事实记录。

### 数据架构原则

- Blueprint Version 是不可变契约。
- Business Instance 绑定创建时的 Blueprint Version。
- 自动执行和协同流程都必须生成 Business Instance。
- Skill Run、Human Task、Acceptance Result 都必须产生 Audit Evidence。
- 报表不是额外人工整理，而是从业务实例和审计证据生成。

## 4. 产品核心功能与竞争力

```mermaid
mindmap
  root((AI-native 企业业务运行框架))
    目标驱动
      输入业务目标
      不以表单为中心
      自动判断执行路径
    双路径执行
      自动执行路径
        Skill 直接完成
        AI 验收
      协同流程路径
        动态表单
        人工审批
        权限隔离
      混合路径
        AI 预处理
        人工确认
    统一上下文
      数据
      规则
      Workflow
      权限
      审计证据
      AI 可连接上下文
    可审计
      人工决定
      Skill 调用
      数据变化
      验收结果
    可演进
      LocalStorage 到数据库
      Mock AI 到真实 Agent
      轻量 Workflow 到流程引擎
      单场景到多场景
```

### 核心竞争力

1. **从表单驱动升级为目标驱动**
   用户输入业务目标，系统决定如何完成，而不是让用户先设计表单。

2. **AI 是执行者和验收者**
   AI 不只是生成建议或聊天回复，还可以通过 Skill 执行业务动作，并参与验收。

3. **自动执行与人工协同共用同一套框架**
   简单事务走 Skill，复杂事务走表单和 Workflow，最终都沉淀为业务实例。

4. **统一业务上下文对 AI 友好**
   数据、规则、权限、流程状态和审计证据都结构化，AI 不需要临时拼接碎片化接口。

5. **天然支持审计和复盘**
   每次人工动作、Skill 调用、数据变化和验收结论都可以成为证据。

6. **具备从原型到生产的替换路径**
   当前本地存储、确定性生成器和轻量 Workflow 都可以逐步替换为数据库、真实 AI Agent 和生产级流程引擎。

## 5. 当前与目标架构的差距

| 能力 | 当前已有 | 目标状态 |
| --- | --- | --- |
| 非结构化输入 | 项目审计描述生成 Blueprint | 任意业务目标进入 Intent Intake |
| 任务分发 | 暂无独立 Task Routing Skill | 自动判断执行路径 |
| 自动执行 | AI CEO 建议雏形 | Skill Runtime 完成业务交付 |
| 协同流程 | 项目审计表单和审批流已跑通 | 多业务场景可配置生成 |
| 权限隔离 | 前端角色视图和字段权限 | 真实用户、组织和后端权限 |
| 数据存储 | LocalStorage | 数据库与审计证据库 |
| AI 验收 | 尚未独立成模块 | Acceptance Module 调用 Skill 验收 |
| 报表 | 初步报表和建议 | 基于业务实例的通用报表 |

## 6. Workflow、Skill、LangGraph 与数据库技术方案

下一阶段采用渐进式产品架构：先保持模型清晰，再逐步替换生产基础设施。核心原则是不让 Workflow、Skill、数据库各自独立发展，它们都必须围绕 Business Operating Blueprint、Business Instance 和 Audit Evidence 协作。

```mermaid
flowchart TD
  A["Business Operating Blueprint<br/>唯一中心契约"] --> B["Execution Plan<br/>执行计划"]

  B --> C["Workflow Runtime<br/>组织流程 / 人工任务 / 审批"]
  B --> D["Skill Runtime<br/>自动执行 / AI 工具调用"]

  C --> C1["XState + Postgres 状态持久化<br/>MVP 阶段"]
  C --> C2["Temporal<br/>生产长流程 / 重试 / 定时 / 跨系统"]

  D --> D1["Simple Skill<br/>普通函数或内部服务"]
  D --> D2["LangGraph Skill<br/>复杂 Agent 编排"]
  D --> D3["External Tool Skill<br/>MCP / 企业 API / 脚本"]

  D2 --> E["LangGraph"]
  E --> E1["多步骤推理"]
  E --> E2["工具调用"]
  E --> E3["条件分支"]
  E --> E4["Human-in-the-loop 中断"]
  E --> E5["Checkpoint / Resume"]

  C1 --> F["Business Instance"]
  C2 --> F
  D1 --> F
  D2 --> F
  D3 --> F
  F --> G["Supabase Postgres<br/>业务实例 / 证据 / 报表"]
  G --> H["Audit Evidence<br/>Skill Run / Human Task / Acceptance Result"]
```

### 6.1 Workflow 推荐

**MVP 阶段推荐：XState / 轻量 Workflow Runtime + PostgreSQL 状态持久化。**

当前业务模型仍在演进，最重要的是先把 Business Goal、Task Routing、Business Operating Blueprint、Business Instance 和 Acceptance Module 的边界跑清楚。XState 适合表达明确状态机，便于继续保持当前 TypeScript 测试方式。

**生产增强阶段推荐：Temporal。**

当流程需要跨天运行、失败重试、定时任务、跨系统补偿、外部 API 长事务时，再引入 Temporal。Temporal 管理可靠执行，但不应替代 Business Operating Blueprint，也不应成为业务语义的唯一来源。

推荐边界：

```text
Workflow Runtime 管组织流程：
- 人工任务
- 审批和驳回
- SLA / 超时
- 流程状态
- 业务实例状态推进
```

### 6.2 Skill / Agent 推荐

**推荐：自定义 SkillRegistry + Skill Runtime，OpenAI / LangGraph / MCP / 企业 API 都作为适配器。**

Skill 在本产品中是业务执行单元，不是无审计的工具函数。每次 Skill 调用都必须形成 Skill Run，并写入 Audit Evidence。

推荐 Skill 契约：

```ts
type SkillDefinition = {
  id: string
  name: string
  description: string
  inputSchema: unknown
  outputSchema: unknown
  requiredPermissions: string[]
  auditPolicy: 'always' | 'on_failure' | 'manual'
}
```

### 6.3 LangGraph 推荐位置

**推荐：LangGraph 放在复杂 Skill 内部，作为 Agent Orchestration 层。**

LangGraph 适合处理 AI 内部的多步骤推理、工具调用、条件分支、检查点恢复和 Human-in-the-loop 中断。它不应该成为企业审批流的主 Workflow 引擎，也不应该成为 Business Instance 的主状态账本。

正确关系：

```text
Business Operating Blueprint
→ Skill Runtime
→ LangGraph Skill
→ LangGraph 内部多步骤 Agent 编排
→ Skill Run Output
→ Acceptance Module
→ Business Instance / Audit Evidence
```

不推荐关系：

```text
Business Operating Blueprint
→ LangGraph 直接管理所有人工审批、企业流程状态和业务实例主状态
```

原因：

- 企业审批需要强权限、强审计和稳定状态模型。
- Human Task、审批记录和业务实例状态应由 Workflow Runtime 与数据库共同管理。
- LangGraph 的 Human-in-the-loop 更适合 AI 执行过程中的暂停、补充和继续，而不是作为组织审批主账本。

### 6.4 数据库推荐

**推荐：Supabase Postgres。**

原因：

- 底层是标准 PostgreSQL，迁移路径清晰。
- JSONB 适合保存 Blueprint Version、业务实例当前数据和 Skill 输出。
- Supabase Auth 可以较快补齐真实登录。
- Row Level Security 可以支撑组织、角色和业务实例范围的数据隔离。
- 适合从当前 LocalStorage Adapter 渐进迁移，不需要一次性重写所有业务模块。

推荐首批数据表：

```text
organizations
users
memberships

business_goals
blueprint_versions
business_instances

execution_steps
human_tasks
skill_runs
acceptance_results

audit_evidence
report_snapshots
```

## 7. 是否需要修改之前的代码和架构

**需要修改，但不建议一次性大改。**

当前代码已经证明“项目审计协同流程”可行，但产品架构已经升级为“业务目标驱动 + 双路径执行 + AI 验收 + 业务实例沉淀”。因此之前代码需要从项目审计专用结构，逐步抽象成 Business Operating Blueprint 的通用结构。

### 7.1 必须调整的方向

| 调整项 | 为什么要改 | 推荐方式 |
| --- | --- | --- |
| `demo/` 命名 | 产品定位不再是 demo | 后续迁移为 `app/` 或 `console/` |
| `demo-scenario` 命名 | “演示数据”不再是中心语言 | 改为 `seed`、`starter-data` 或 `workspace-bootstrap` |
| `ProjectAuditBlueprint` | 目前仍是项目审计专用蓝图 | 新增 `BusinessOperatingBlueprint` 通用类型，再把项目审计作为 scenario |
| `audit-*` runtime | 业务逻辑仍围绕项目审计 | 提取通用 `business-instance`、`execution-step`、`human-task` |
| `ai-ceo-assessment` | 只覆盖项目审计建议 | 抽象成 `acceptance` / `assessment` 模块 |
| `LocalStorageAdapter` | 只能单人本地测试 | 保持接口，新增 Supabase/Postgres Adapter |
| 前端角色切换 | 不是真实权限 | 下一阶段引入 Auth、User、Role、Organization |

### 7.2 推荐的重构顺序

```mermaid
flowchart TD
  A["第 1 步：命名产品化<br/>demo → app / console"] --> B["第 2 步：新增通用 Blueprint 类型<br/>BusinessOperatingBlueprint"]
  B --> C["第 3 步：新增 Task Routing Skill<br/>目标分发"]
  C --> D["第 4 步：新增 Business Instance 数据模型<br/>统一自动执行和协同流程"]
  D --> E["第 5 步：抽象 Skill Runtime<br/>Simple Skill / LangGraph Skill / External Tool Skill"]
  E --> F["第 6 步：抽象 Acceptance Module<br/>验收结果入库"]
  F --> G["第 7 步：接 Supabase Postgres<br/>Auth + RLS + 持久化"]
  G --> H["第 8 步：保留项目审计为第一个场景<br/>迁移到通用框架之上"]
```

### 7.3 下一批可开发 Issue 建议

1. **产品命名迁移**：将 `demo/`、`Demo`、`demo-scenario` 等命名迁移为产品化命名。
2. **BusinessOperatingBlueprint 类型**：定义通用蓝图契约，兼容自动执行和协同流程。
3. **Task Routing Skill 原型**：输入 Business Goal，输出 routeType、routingReason、requiredData、riskLevel。
4. **Business Instance Store**：新增业务实例中心模型，统一承接 Human Task 和 Skill Run。
5. **Skill Runtime 契约**：定义 SkillDefinition、SkillRun、Skill Adapter。
6. **LangGraph Skill Adapter 设计**：明确复杂 Skill 如何包装 LangGraph，并如何输出审计证据。
7. **Acceptance Module**：定义 passed、needs_human_review、failed 三类验收结果。
8. **Supabase Postgres 数据库迁移方案**：先设计表结构和 RLS，再替换 LocalStorage。
