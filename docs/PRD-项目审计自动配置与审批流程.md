# 项目审计自动配置与审批流程 PRD

## 问题陈述

企业需要周期性审计项目是否持续符合战略和业务目标，重点检查里程碑完成情况、战略目标完成进度，以及成本、预算等执行表现。现实中，这类审计往往依赖人工创建表格、重复录入项目资料、通过邮件或聊天工具催办审批，并且难以保留项目数据被修改的完整证据。

审计管理员希望通过一段业务描述，快速生成可预览、可部署、可运行的项目审计业务蓝图，而不是每次重新开发表单、权限和审批页面。项目负责人需要在预填项目资料的基础上完成本周期审计，并对申报内容进行正式确认；分管项目 VP 和 AI CEO 需要按顺序审批，并能够查看原始项目快照、当前申报值和完整变更记录。

当前系统已经验证了采购场景中的业务蓝图、动态表单、轻量流程、统一存储和 Agent Activity，但现有业务蓝图仍以采购字段和角色为主，尚不具备通用实体模型、预填规则、版本管理、实例可见范围、字段权限、可重复表单分组、退回重审和周期性批量实例生成能力。

第一阶段需要完成一个可实际演示的项目审计纵向链路，同时保持底层契约领域无关，为未来扩展到合同、预算、采购等业务场景留下稳定边界。

这个阶段还需要验证一个更重要的系统目的：AI 与人能够共享同一份项目审计上下文。AI 不应通过抓取页面或调用大量彼此割裂的接口来猜测项目情况，而应在权限范围内直接使用业务蓝图、项目快照、当前申报值、适用规则、流程状态、变更记录和审批记录。AI 的建议、人工决定和后续修改都必须回到反馈与审计闭环中。

## 解决方案

建设“项目审计自动配置与审批流程”第一阶段能力：

1. 审计管理员输入项目审计业务描述。
2. 确定性的 `ProjectAuditBlueprintGenerator` 将描述转换为通用、版本化的业务蓝图草稿。
3. 管理员预览实体模型、表单、预填规则、角色、权限和审批流程后部署业务蓝图。
4. 管理员在本地项目台账中维护项目基础资料，并创建审计周期、选择审计范围。
5. 启动审计周期后，系统为每个选中项目生成一条审计实例，并复制项目快照作为预填数据。
6. 项目负责人可以修改预填数据，系统保存字段级变更记录，但不自动回写项目台账。
7. 流程依次经过项目负责人自审批、分管项目 VP 审批和 AI CEO 审批。
8. VP 或 AI CEO 驳回后，实例进入“退回修改”，由项目负责人修改并重新走完整审批流程。
9. 系统通过角色切换器模拟身份，并按业务蓝图计算实例可见范围和字段访问权限。
10. 第一阶段使用本地统一存储，不接入真实数据库、真实认证或真实 AI。
11. Mock AI CEO 与人工审批界面使用同一份结构化审计上下文，验证未来真实 AI 可以替换建议模块，而不需要重新为 AI 建造一套业务接口。

完整业务链路：

```text
业务描述
→ 生成业务蓝图草稿
→ 预览并部署业务蓝图
→ 维护项目台账
→ 创建审计周期并选择项目
→ 启动周期并生成审计实例
→ 项目负责人填写和自审批
→ 分管项目 VP 审批
→ AI CEO 审批
→ 已通过
```

## 实现决策

### 1. 产品和范围

- 第一阶段只保证项目审计场景端到端运行，不承诺处理任意业务描述。
- 项目审计是首个支持场景，不是系统边界；采购、合同、预算等未来场景应复用同一通用运行时。
- 现有采购 demo 作为架构和测试先例保留，不作为本阶段主要用户流程。
- 第一阶段继续采用静态 Web demo，不引入后端服务。

### 2. 通用业务蓝图

- 将现有业务蓝图演进为通用、版本化的 `Blueprint V2`。
- 业务蓝图至少包含元信息、实体模型、表单 Schema、预填规则、角色、实例可见范围、字段访问规则、流程定义和变更审计规则。
- 业务蓝图具有“草稿”和“已部署”生命周期。
- 已部署版本不可直接覆盖；修改配置必须形成新版本。
- 审计实例保存其使用的业务蓝图版本标识，后续新版本不得改变既有实例。
- 业务蓝图不得包含 React 组件或页面级条件判断。

### 3. 项目审计蓝图生成器

- 建立独立的 `ProjectAuditBlueprintGenerator` 深模块。
- 输入是业务描述，输出是符合通用契约的项目审计业务蓝图草稿。
- 第一阶段使用确定性解析和固定领域规则，不接入真实大模型。
- 生成器负责项目审计默认实体、字段、角色、权限和流程定义，运行时只解释生成结果。
- 未来真实 AI 生成器必须输出同一业务蓝图契约。

### 4. 项目台账

- 建立独立项目台账模块，维护项目当前正式资料。
- 项目包括项目编号、名称、负责人、分管 VP、部门、计划起止日期、战略目标、批准预算、当前成本和里程碑。
- 提供三个预置示例项目。
- 项目台账是审计实例预填数据来源，但不是审计实例中的可变引用。

### 5. 审计周期

- 建立审计周期服务，负责草稿、启动、关闭和审计范围管理。
- 启动周期时批量为范围内项目创建审计实例。
- 以“审计周期标识 + 项目标识”作为业务唯一性约束。
- 启动后允许追加项目；已经开始填写的项目不能移除。
- 关闭周期后不再允许追加项目或创建新实例。

### 6. 审计实例和项目快照

- 建立审计实例工厂，统一处理项目、周期、项目快照、业务蓝图版本和初始状态。
- 项目快照在实例创建时深复制，项目台账之后的变化不得自动传播。
- 项目负责人可以修改实例内快照和审计字段。
- 实例修改不自动回写项目台账。

### 7. 动态表单

- 扩展动态表单引擎，支持单行文本、数字、多行文本、日期、选择、百分比、金额和可重复分组。
- 项目审计表单由业务蓝图动态生成，不在页面中硬编码字段结构。
- 里程碑和战略目标采用可重复分组。
- 表单按区域组织：基本信息、里程碑、战略目标、执行表现、风险问题、整改计划和负责人总结。
- 预填规则负责从项目快照映射初始字段值。

### 8. 变更审计

- 建立独立的变更审计服务，比较保存前后的审计数据并产生字段级记录。
- 每条记录包含修改人、修改时间、字段路径、原值、新值和修改原因。
- 项目编号、项目负责人、分管项目 VP、批准预算和战略目标属于关键字段，修改时原因必填。
- 变更记录为追加式，不允许覆盖或删除。
- 审批人和全局查看者能够查看完整记录。

### 9. 权限

- 建立独立权限计算器，同时处理实例可见范围和字段访问规则。
- 第一阶段通过角色切换器模拟当前用户，不实现身份认证。
- 角色包括审计管理员、项目负责人、分管项目 VP、AI CEO 和全局查看者。
- 字段权限支持不可见、只读和可编辑。
- 权限结果由角色、用户与项目关系、流程阶段和业务蓝图共同决定。
- UI 只能消费权限结果，不得自行复制业务权限规则。

### 10. 流程运行时

- 将现有只支持顺序批准或终止拒绝的流程运行时演进为可配置状态机。
- 项目审计流程为：填写、负责人自审批、VP 审批、AI CEO 审批、已通过。
- 驳回是审批决定记录，不是审计实例终态。
- VP 或 AI CEO 驳回后，实例进入退回修改，并从项目负责人自审批重新发起完整流程。
- 项目负责人可以在 VP 尚未审批前撤回实例。
- 通过意见可选，驳回意见必填。
- 每个审批动作保存角色、操作者、决定、意见和时间。

### 11. AI CEO 审计建议

- 建立独立的 Mock AI CEO 审计建议模块。
- 根据里程碑延期、战略进度、预算偏差、成本超支、风险问题和变更记录生成结果。
- 输出风险等级、关键发现、建议结论和建议理由。
- AI CEO 审批仍由演示用户点击确认，不进行无人工确认的自动终审。
- 未来真实 AI 实现应替换建议生成模块，不改变流程和 UI 契约。

### 12. 统一存储

- 扩展现有统一存储适配器，保存业务蓝图版本、项目台账、审计周期、审计范围、审计实例、项目快照、变更记录、审批记录和 AI CEO 审计建议。
- UI 不得直接读写浏览器存储。
- 第一阶段继续使用 LocalStorage 实现，并保留内存实现供测试使用。
- 提供可二次确认的演示数据重置操作。

### 13. 控制台交互

- 在现有框架控制台中增加项目审计自动配置、项目台账、审计周期和项目审计 Runtime 体验。
- 自动配置页面支持描述输入、生成、预览和部署。
- 项目台账页面支持查看、新增和编辑项目。
- 审计周期页面支持草稿创建、项目选择、启动、追加项目和关闭。
- Runtime 页面根据当前模拟角色展示可见实例、可执行操作和字段权限。
- 审计详情展示项目快照、当前申报值、变更记录、审批记录和 AI CEO 建议。

### 14. 深模块边界

需要优先保持简单公共接口、复杂内部实现的深模块包括：

- 业务蓝图生成器：业务描述输入，业务蓝图草稿输出。
- 审计实例工厂：周期、项目、业务蓝图版本输入，审计实例输出。
- 动态表单解释器：表单 Schema 和数据输入，可渲染字段模型与校验结果输出。
- 变更审计服务：原数据、新数据和操作者输入，变更记录或校验错误输出。
- 权限计算器：用户、角色、实例和流程阶段输入，可见性与字段访问结果输出。
- 项目审计流程运行时：实例与业务动作输入，新状态和审批记录输出。
- AI CEO 建议生成器：审计实例和历史证据输入，结构化建议输出。
- 存储适配器：稳定的数据读写端口，当前实现为 LocalStorage。

## 测试决策

### 1. 测试原则

- 使用 TDD，一次只验证一个外部可观察行为。
- 测试公共接口和业务结果，不测试私有函数、内部调用次数或 React 组件内部状态。
- 核心模块采用集成式行为测试，尽量使用真实领域对象和内存存储适配器。
- 时间、标识和示例数据通过可注入方式固定，保证测试可重复。
- UI 测试聚焦关键用户路径，不追求每个样式和文案的自动化覆盖。

### 2. 需要重点测试的模块

- 业务蓝图生成器：项目审计描述能够生成完整、稳定且可部署的通用业务蓝图。
- 业务蓝图版本：已部署版本不可覆盖，旧实例继续绑定旧版本。
- 项目台账：项目资料可以保存、读取和更新。
- 审计周期服务：状态转换、审计范围、追加项目和关闭规则正确。
- 审计实例工厂：启动周期批量创建实例、预填快照以及同周期项目唯一性正确。
- 动态表单解释器：字段类型、可重复分组、必填校验和预填映射正确。
- 变更审计服务：字段差异、关键字段原因校验和追加式记录正确。
- 权限计算器：五种角色的实例可见范围和字段访问权限正确。
- 流程运行时：自审批、VP 审批、AI CEO 审批、撤回、驳回、退回修改和重新审批正确。
- AI CEO 建议生成器：不同项目风险组合生成稳定的风险等级、发现和建议。
- 存储适配器：所有新实体持久化、读取和重置行为正确。

### 3. 关键端到端路径

- 业务描述生成蓝图、预览、部署并创建新版本。
- 管理员维护项目、创建周期、选择三个项目并启动。
- 系统为三个项目创建独立实例并预填项目快照。
- 项目负责人修改关键字段、填写原因并完成自审批。
- VP 查看变更记录并批准，流程进入 AI CEO。
- AI CEO 查看审计建议并批准，实例进入已通过。
- VP 驳回后退回项目负责人，修改后重新走完整流程。
- AI CEO 驳回后退回项目负责人，修改后重新经过 VP 和 AI CEO。
- 不同角色切换后只能看到允许的实例和字段。
- 刷新页面后数据和流程状态保持不变。
- 重置演示数据后恢复三个示例项目和初始配置。

### 4. 现有测试先例

- 现有项目已经使用 Vitest 对业务蓝图、申请创建、风险分析、审批路由、流程状态、统一存储、Agent Activity 和报表生成进行行为测试。
- 新模块继续沿用内存存储适配器和固定时间、固定标识的测试方式。
- 新流程运行时不能以现有“拒绝即终止”的测试作为项目审计语义；应新增退回修改和重新审批行为测试。

### 5. 交付验证

- 完整自动化测试通过。
- TypeScript 编译和 Vite 生产构建通过。
- 浏览器中可以完成一条项目审计主流程。
- 浏览器中可以验证一次 VP 驳回和重新审批。
- 浏览器中可以切换五种角色检查权限。
- 浏览器刷新后数据保持。

- 真实组织架构和人员同步。
- 真实 OpenAI、本地模型或其他 AI API。
- AI CEO 无人工确认的自动最终审批。
- 邮件、飞书、企业微信、Slack 或短信通知。
- 外部项目管理、财务、ERP、CRM 或 OA 系统集成。
- 数据看板、管理驾驶舱、BI 报表和趋势分析。
- 项目台账变更的正式审批流程。
- 审计实例自动回写项目台账。
- 附件上传、电子签名和打印归档。
- 多租户、生产级权限隔离和安全合规能力。

- 可视化拖拽表单设计器或 BPMN 流程设计器。

## 补充说明

- 本 PRD 使用 `CONTEXT.md` 中的统一领域词汇。
- “ICO 审批”是误写，所有文档和界面统一使用“AI CEO 审批”。
- 驳回是一条审批决定，项目审计实例随后进入“退回修改”，不是进入不可恢复的“已驳回”终态。
- 项目台账代表当前正式资料；项目快照代表某次审计创建时的数据；当前申报值代表项目负责人在该审计实例中修改后的数据，三者不得混用。
- 第一阶段成功标准不是界面数量，而是从业务描述到业务蓝图、周期、实例、表单、权限和审批的完整链路能够实际运行。
- 完成本 PRD 后，下一步应使用 `to-issues` 将其拆分为可独立开发、带依赖关系和验证方式的 GitHub Issues。
